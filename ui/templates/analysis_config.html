How this works

  • By default, Analysis Service includes this prompt in every batched call to Azure AI Foundry/ML for this tenant, alongside the records being analyzed.
  • Choosing "OpenAI-compatible" switches this tenant to the standard /v1/chat/completions API instead — one client covers Ollama, OpenAI, and Azure OpenAI in compatible mode, called once per record rather than batched.
  • Changes take effect within seconds — Config/Admin Service publishes the change over the message bus and Analysis Service picks it up automatically, no restart needed.
  • Trigger Engine still decides when to actually create an Event: it evaluates your Triggers against whatever numeric fields the AI returns, so a prompt that changes what the AI reports may also change which Triggers fire.
  • Every save is written to an immutable audit log for compliance review — the API key is redacted before it reaches that log, but is otherwise stored as entered (config-as-data convention; see ADR-0031).
Resilience posture

Transient model failures do not stop the pipeline.

If an OpenAI-compatible model is overloaded, rate-limited, times out, or returns a temporary upstream failure, the batch is attempted once against the platform default Foundry model. Foundry-primary deployments can also configure a separate fallback model at service deployment time. Only after both attempts fail does the normal retry and dead-letter policy take over.
